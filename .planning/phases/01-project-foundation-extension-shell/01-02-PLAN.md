Create `packages/extension/wxt.config.ts`:
```ts
import { defineConfig } from 'wxt';
import tailwindcss from '@tailwindcss/vite';

export default defineConfig({
  modules: ['@wxt-dev/module-react'],
  vite: () => ({
    plugins: [tailwindcss()],
  }),
  manifest: {
    name: 'Inspatch',
    description: 'Visual code editing for locally-served web pages',
    permissions: ['activeTab', 'sidePanel', 'scripting'],
    host_permissions: ['http://localhost:*/*'],
    action: {},
  },
});
```

- `modules: ['@wxt-dev/module-react']` adds `@vitejs/plugin-react` to Vite and configures JSX transform automatically.
- `@tailwindcss/vite` plugin enables Tailwind CSS v4 with automatic content detection.
- `permissions` per D-05: `activeTab` (tab access on click), `sidePanel` (side panel API), `scripting` (content script injection for future phases).
- `host_permissions: ['http://localhost:*/*']` restricts to local dev servers only.
- `action: {}` is REQUIRED — without it, `openPanelOnActionClick` has no toolbar icon to attach to.

Create `packages/extension/entrypoints/background.ts`:
```ts
export default defineBackground(() => {
  chrome.sidePanel.setPanelBehavior({ openPanelOnActionClick: true })
    .catch(console.error);
});
```

Per D-06 and CP-6 mitigation: `setPanelBehavior({ openPanelOnActionClick: true })` binds the side panel to the toolbar icon click. This bypasses the fragile gesture window problem with `sidePanel.open()` (CP-6: gesture expires in ~1ms after async work). The `.catch()` handles environments where the API is unavailable. `defineBackground` is auto-imported by WXT. ALL runtime code MUST be inside this callback per WXT entrypoint conventions (research finding #7: background modules are imported in Node during build).

Create `packages/extension/entrypoints/content.ts`:
```ts
export default defineContentScript({
  matches: ['http://localhost:*/*'],
  main() {
    console.log('[Inspatch] Content script loaded');
  },
});
```

Content script matches only `http://localhost:*/*` per D-05 host permissions scope. `defineContentScript` is auto-imported by WXT. This is a minimal placeholder — real element selection logic is added in Phase 2. All runtime code is inside `main()` per WXT entrypoint rules.

After creating all three files, run `cd packages/extension && bunx wxt prepare` to generate the `.wxt/` directory (needed for TypeScript resolution via the extension's tsconfig.json which extends `.wxt/tsconfig.json`).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Extension manifest → Chrome | Permission declarations gate what the extension can access |
| Content script → page DOM | Content script runs in isolated world on localhost pages |
| Side panel → extension runtime | Side panel is a separate document communicating via chrome.runtime |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04 | Elevation of Privilege | manifest permissions | mitigate | Minimal permissions per D-05: activeTab (not <all_urls>), sidePanel, scripting only; host_permissions restricted to http://localhost:*/* |
| T-01-05 | Spoofing | content script matches | mitigate | Content script matches only http://localhost:*/* — cannot run on arbitrary websites; prevents accidental data capture on non-dev pages |
| T-01-06 | Information Disclosure | console.log in content script | accept | Debug logging only in Phase 1; production builds will strip console.log; local-only tool with no sensitive data at this stage |
</threat_model>
